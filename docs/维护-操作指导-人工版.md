# 维护-操作指导（人工版）

## 1. 当前系统概况

- 代码仓库：`origin` 指向 GitHub 私有仓库 `zc63463-cmyk/vocab-observatory`
- 默认分支：`main`
- 生产站点：`https://vocab-observatory.vercel.app`
- 部署方式：GitHub `main` 分支推送后，Vercel 自动触发生产部署
- 数据层：Supabase
- 登录方式：Owner 邮箱 + Magic Link
- 内容源：GitHub 仓库 `zc63463-cmyk/Obsidian-Eg`

## 2. 日常维护入口

### 2.1 看线上站点是否正常

优先检查：

- 首页：`https://vocab-observatory.vercel.app`
- 词条库：`https://vocab-observatory.vercel.app/words`
- 仪表盘：`https://vocab-observatory.vercel.app/dashboard`

如果公开页打不开，优先看：

- Vercel 项目状态
- 最近一次部署是否成功
- Vercel 环境变量是否被误删

如果只有 owner 页面打不开，优先看：

- Supabase Auth 回调地址
- Owner 邮箱是否被改动
- 登录状态是否失效

### 2.2 看最近一次同步是否正常

进入 `/dashboard` 查看：

- 最近一次同步时间
- created / updated / unchanged / soft deleted
- error count
- 最近错误摘要

如果 dashboard 显示“尚未应用导入追踪迁移”，说明 Supabase 还没执行 `0003_import_tracking.sql`。

## 3. 内容同步操作

### 3.1 生产环境手动同步

当 `Obsidian-Eg` 有更新，或者你怀疑 Vercel Cron 没跑时，手动触发：

```bash
curl -X POST https://vocab-observatory.vercel.app/api/imports/github \
  -H "Authorization: Bearer <IMPORT_SECRET>"
```

执行后去 `/dashboard` 看导入结果。

### 3.2 本地同步

本地开发或排查内容问题时使用：

```bash
npm run sync:vault
```

用途：

- 验证解析器是否还能处理上游 markdown
- 看新增词条是否能正常落库
- 检查失败文件与失败原因

### 3.3 同步异常时怎么判断

常见现象和处理：

- 导入成功但词条数量不变
  - 说明大多是 `unchanged`
  - 看 dashboard 的 `created / updated / unchanged`
- 导入失败文件增加
  - 去 dashboard 看 `import_errors`
  - 优先检查 frontmatter 是否非法、字段是否有未加引号的冒号、正文标题是否缺失
- 线上同步失败，本地成功
  - 看 Vercel 环境变量是否一致
  - 看 `IMPORT_SECRET`、`SUPABASE_SERVICE_ROLE_KEY` 是否有效

## 4. 代码发布流程

### 4.1 本地开发前

先拉最新代码：

```bash
git pull origin main
```

### 4.2 提交与上线

```bash
git add -A
git commit -m "你的变更说明"
git push origin main
```

推送完成后：

- Vercel 会自动开始生产部署
- 去 Vercel Dashboard 查看 build / deploy 状态
- 部署成功后，刷新生产站点确认结果

### 4.3 发布前固定检查

本地至少执行：

```bash
npm run lint
npm run typecheck
npm run test
npm run build
```

如果是内容同步、Auth、review 逻辑相关改动，发布后还要补做人工验收：

- `/words`
- `/words/[slug]`
- `/dashboard`
- `/review`
- `/auth/login`

## 5. 多设备访问说明

当前正确访问方式是：

- 任意设备打开 `https://vocab-observatory.vercel.app`

不要再依赖：

- `127.0.0.1:3000`
- 本机开发服务器

这些地址只适用于本地调试。

## 6. Owner 登录维护

### 6.1 当前登录配置

- Owner 邮箱由 `OWNER_EMAIL` 控制
- Magic Link 发信由 Supabase Auth + Resend SMTP 支持

### 6.2 登录异常优先检查

1. Supabase `Authentication -> URL Configuration`
   - `Site URL=https://vocab-observatory.vercel.app`
   - Redirect 中包含 `https://vocab-observatory.vercel.app/auth/callback`
2. Vercel 环境变量中的 `NEXT_PUBLIC_SITE_URL`
3. Resend / SMTP 是否仍可发信
4. `OWNER_EMAIL` 是否被改动

### 6.3 更换 owner 邮箱

需要同时改两处：

- 本地 `.env.local`
- Vercel 项目的 `OWNER_EMAIL`

改完后重新部署，再用新邮箱测试一次 magic link。

## 7. 数据库迁移维护

当前关键迁移：

- `0001_init.sql`
- `0002_rls.sql`
- `0003_import_tracking.sql`

新增迁移时要求：

- 不修改已执行迁移文件的历史语义
- 新增新文件，例如 `0004_xxx.sql`
- 先在 Supabase 执行，再更新 README / 维护文档

## 8. 安全注意事项

以下内容不能提交进 Git：

- `.env.local`
- Supabase service role key
- GitHub PAT
- Resend API key
- `IMPORT_SECRET`
- `CRON_SECRET`

如果密钥曾经在聊天记录、截图、日志里暴露，优先轮换：

1. GitHub PAT
2. Supabase `SUPABASE_SERVICE_ROLE_KEY`
3. Resend API key
4. `IMPORT_SECRET`
5. `CRON_SECRET`

## 9. 推荐维护节奏

### 每次改代码后

- 跑本地四项检查
- 推送 `main`
- 看 Vercel 自动部署
- 生产站抽查关键页面

### 每次内容源更新后

- 触发一次生产导入
- 看 dashboard 的导入状态
- 抽查 1 到 3 个词条页

### 每月一次

- 检查 Vercel 环境变量是否完整
- 检查 Supabase Auth 回调配置
- 检查 Resend 发信是否正常
- 轮查最近导入失败文件

## 10. 遇到问题时的最短排查顺序

1. 先确认是本地问题还是生产问题
2. 再区分是页面问题、登录问题、导入问题还是部署问题
3. 最后再看数据库或外部服务

推荐顺序：

1. Vercel 最近一次部署是否成功
2. 生产域名是否可访问
3. `/api/words` 是否正常返回
4. `/api/imports/github` 是否能手动触发
5. Supabase Auth / SMTP 是否正常
